# Placement spacing halo for coal/gas/wind

Status: ready-for-agent
Category: enhancement

## Parent

[PRD: Economy & balance pass](../PRD.md)

## What to build

Introduce a one-cell no-build halo around coal, gas, and wind generators so that power facilities take real spatial planning instead of clustering into a single complex. Roads and batteries are admitted inside the halo (so plants can still be serviced and storage co-located). Solar farms and batteries have no spacing rule of their own — renewable-and-storage clusters remain viable. Existing tiles violating the new rule are grandfathered; the rule is enforced only at build time.

Implementation:

- New `placement` module owning the spacing matrix as a single declarative table. Pure-function surface: `validate(candidate_type, candidate_coords, tiles) -> offending_neighbor | None`. Halo entries: coal, gas, wind. Admitted neighbors inside any halo: roads and batteries.
- World's `build` entry point consults this module after road-adjacency but before treasury debit.
- New structured error code (analogous to `no_road_adjacency`) for spacing rejection. The contract is the code plus the offending neighbor coordinates; the UI owns the user-facing message.
- Catalog descriptions for coal, gas, and wind reflect the spacing halo.

## Acceptance criteria

- [ ] `placement.validate` returns the offending neighbor at distance 1 (8-neighborhood) for halo'd types adjacent to other non-admitted tiles.
- [ ] Returns `None` at distance ≥ 2.
- [ ] Returns `None` when the only neighbor inside the halo is a road or battery.
- [ ] Solar and battery candidates impose no halo on neighbors.
- [ ] When a candidate violates spacing against two existing tiles, exactly one offending neighbor is surfaced deterministically.
- [ ] Existing tiles violating the rule at release are grandfathered (no re-validation on load; no bulldoze pass).
- [ ] Build endpoint emits the new structured error code with the offending neighbor coordinates; existing error codes (e.g. `no_road_adjacency`) are unchanged.
- [ ] Coal, gas, and wind catalog descriptions mention the one-cell halo.
- [ ] `make check` passes.

## Agent Brief

**Category:** enhancement
**Summary:** Enforce a one-cell no-build halo around coal, gas, and wind generators at build time, with roads and batteries admitted inside the halo.

**Current behavior:**
Generation facilities can be packed cell-by-cell into a single sprawling complex. There is no spatial planning trade-off for power.

**Desired behavior:**
A new placement validator owns a declarative spacing matrix and a pure-function surface: given a candidate tile type, candidate coordinates, and the full set of existing tiles, it returns either the offending neighbor tile (rejection) or `None` (allowed). Halo entries: coal, gas, wind. Admitted neighbors inside any halo: roads and batteries. Solar and battery placements impose no halo of their own. The build entry point calls this validator after road-adjacency but before treasury debit. Rejections surface a new structured build-error code (peer to `no_road_adjacency`) carrying the offending neighbor's coordinates. Existing tiles that violate the rule at release are grandfathered — validation runs at build time only, never on load. Coal, gas, and wind catalog descriptions mention the one-cell halo.

**Key interfaces:**
- A new placement module exposing `validate(candidate_type, candidate_coords, tiles) -> offending_neighbor | None`. The spacing matrix lives as a single declarative table inside this module.
- The build entry point — orders validation as: road-adjacency → spacing → treasury debit.
- A new structured build-error code for spacing rejection. The contract: code identifier plus the offending neighbor coordinates. UI owns the user-facing string.
- Coal, gas, and wind catalog entries — descriptions mention the halo.

**Acceptance criteria:**
- [ ] For coal/gas/wind candidates, `validate` returns the offending neighbor when any non-road, non-battery tile sits in the 8-neighborhood at distance 1.
- [ ] For coal/gas/wind candidates, `validate` returns `None` at Chebyshev distance ≥ 2.
- [ ] For coal/gas/wind candidates, roads and batteries inside the halo do not trigger rejection.
- [ ] Solar and battery candidates impose no halo on neighbors (their placement is rejected only by other rules).
- [ ] When a candidate violates spacing against multiple existing tiles, exactly one offending neighbor is surfaced deterministically.
- [ ] Existing tiles violating the rule at release continue to load and operate — no bulldoze pass, no re-validation on load.
- [ ] The build endpoint emits the new structured error code with offending-neighbor coordinates; pre-existing error codes (e.g. `no_road_adjacency`) are unchanged in shape and ordering.
- [ ] Coal, gas, and wind catalog descriptions mention the one-cell halo and the road/battery admission.
- [ ] `make check` passes.

**Out of scope:**
- Halo rules for non-generator tiles (housing, commercial, industrial, refinery, wells, etc.).
- Retroactive enforcement on existing tiles.
- UI rendering of the rejection message (contract only).
- Future placement rules (noise zones, water-table) — the matrix is extensible but those entries are not in this issue.
