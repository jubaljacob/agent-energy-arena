# PRD: Open-Source Arena

Status: needs-triage

## Problem Statement

The project began life as a 24-hour hackathon scaffold for a single in-person event. The world simulation, reference agents, scoring, and UI are all in place and working, but the project is shaped around one cohort of participants, one held-out evaluation seed, and one moment in time. To turn it into a durable open-source benchmark — an "arena" where the community can submit agents, compare them on shared scenarios, and stress-test them under adversarial conditions chosen by maintainers — the project needs four things it does not have today:

1. **A way to author and ship stress scenarios** (e.g. "no wind for a week starting day 21, oil price doubled on day 70") so agents can be evaluated under conditions richer than the stochastic events the world already samples.
2. **A daily history of each game on disk in a self-contained format**, so the UI can replay a finished game without re-running the simulation, and so external reviewers can audit any score by reading the file.
3. **Playback controls in the UI** — load a recorded game from disk, scrub through it day by day with play/pause and a fast-play mode.
4. **A clear comparison protocol** — a small set of public scenarios committed to the repo with reference baselines, plus an organizer-controlled hidden eval suite, and a leaderboard derived from mean-rank aggregation across scenarios.

Beyond those four functional gaps, the project needs the standard documentation and licensing scaffolding an open-source benchmark requires: a rules document agent authors can read end-to-end, an API reference, a scenarios authoring guide, a contributors' guide, and an MIT license. The existing design briefs in `docs/` are written from a hackathon-organizer perspective and assume specific knowledge of the event; they will be archived as historical context, with new docs targeted at the new primary audience: external agent authors who arrive at the repo cold.

## Solution

The simulation, scoring formula, and 16-endpoint API stay essentially unchanged. The new capability is added through five small, focused surfaces:

- **Scenarios** are plain Python classes that implement one method, `apply(world, day)`, called once at the start of each simulated day. A scenario writes into a small overrides dict on world state (for transient effects like a wind clip), mutates mutable price fields on world state (for things like an oil price spike), or appends directly to the active-events list (for forced event injections). Scenarios can be attached at game reset for benchmarking or swapped mid-game via a new endpoint for interactive exploration.
- **Daily history** is written by the simulator after every successful step. Each game produces a run folder containing one JSON-lines file with the full world state and a per-day summary, plus a metadata file and a final snapshot. This is the single source of truth for both replay and audit.
- **The UI** gains a "load run" file picker that reads a run folder directly in the browser (no server involvement), plus play/pause and double-speed playback controls. The existing live-play path is unchanged; the new replay path is a separate render mode toggled by the file picker.
- **The arena** is a small Python package with a CLI that runs `(agent, scenario)` pairs as subprocesses, collects results, and produces a ranked leaderboard. Three reference scenarios ship with the repo, each with a committed baseline.
- **Documentation** is reorganized: seven Markdown files at the repository root, plus an archive folder for the original design briefs.

Pricing constants that today live as module-level constants — crude price, refined price, grid retail/export, per-fuel costs, industrial revenue rate, commercial revenue rate, daily tax per capita, blackout penalty — are promoted to mutable fields on world state, following the precedent of the existing mutable carbon price. This refactor unlocks the scenario override taxonomy without introducing new mechanics. Other constants (emissions intensities, dispatch thresholds, hardware specs, event probabilities) stay where they are.

The score formula does not change. Per-scenario baselines are committed to the repo and regenerated by a Makefile target that the CI pipeline calls on PRs touching world or agent code. Arena ranking uses mean rank across scenarios; ties break on mean raw score.

## User Stories

### Agent authors

1. As an agent author, I want to read a single rules document at the root of the repository, so that I can understand the game mechanics without piecing them together from the source code or historical briefs.
2. As an agent author, I want every API endpoint documented with example request and response shapes, so that I can call the API confidently without reading the FastAPI source.
3. As an agent author, I want to fork the repo and develop my agent under the existing personal-workspace directory, so that I can iterate locally without ceremony.
4. As an agent author, I want to run my agent against all public scenarios from a single command, so that I can see how it generalizes before submitting.
5. As an agent author, I want to read the source code of each shipped scenario, so that I understand exactly which stresses I am being tested against.
6. As an agent author, I want my agent's game to be automatically recorded to disk, so that I can replay any run I find interesting without re-executing the agent.
7. As an agent author, I want to submit my finalized agent to the community directory via pull request, so that maintainers can review it and add it to the leaderboard.
8. As an agent author, I want the arena's hidden evaluation seeds to be different from the public ones, so that I cannot overfit to known scenarios and my submission's generalization is tested honestly.
9. As an agent author, I want clear pre-commit gates (`make check`) that match what CI enforces, so that my PR will not be bounced for stylistic issues.

### Scenario authors

10. As a scenario author, I want to express a stress as a single Python class with one method, so that the authoring surface is as small as possible.
11. As a scenario author, I want to set weather overrides (wind, cloud, irradiance) over a date range, so that I can simulate sustained renewable shortfalls.
12. As a scenario author, I want to mutate pricing fields on world state (crude, refined, grid retail/export, per-fuel cost, tax, industrial/commercial revenue rates, blackout penalty), so that I can simulate policy and market shocks.
13. As a scenario author, I want to inject named events (heatwave, plant failure, fuel price shock, demand surprise, regulatory tightening) on specific days with chosen durations, so that I can compose stress packs deterministically rather than relying on the stochastic sampler.
14. As a scenario author, I want my scenario's effects to be visible in the recorded daily state, so that anyone replaying the run can see what my scenario did and when.
15. As a scenario author, I want my scenario to be loadable by dotted module path, so that it can be discovered without a registry file.

### Human players

16. As a human player, I want to load a recorded game from a folder on my disk into the UI, so that I can review a game that I or an agent played earlier.
17. As a human player, I want play and double-speed play buttons in the UI, so that I can watch a recorded game advance automatically at 0.5 second or 0.25 second per day.
18. As a human player, I want to pause playback at any time, so that I can examine the current day's state without it advancing.
19. As a human player, I want the play button to also work for the live world (no recording loaded), so that the same control auto-advances when I am playing manually.
20. As a human player, I want to attach a stress scenario to my live game mid-play through the UI, so that I can explore how the city responds to a stress in real time.
21. As a human player, I want to step back to previous days while reviewing a recorded game, so that I can compare the state before and after an event.
22. As a human player, I want to share a recorded game by zipping its run folder, so that someone else can drop it into their UI and see the same game I saw — without running the server.
23. As a human player, I want my UI sessions to be recorded the same way agent runs are, so that I can replay my own attempts and learn from them.
24. As a human player, I want existing controls (Next Day, Reset) to keep working unchanged, so that my muscle memory is not broken by the new playback features.

### Maintainers and benchmarkers

25. As a maintainer, I want a hidden evaluation suite I can run periodically against all community agents, so that the public leaderboard reflects performance on scenarios the agents have not seen.
26. As a maintainer, I want to regenerate the leaderboard with one command, so that updating it after merging a new agent is trivial.
27. As a maintainer, I want CI to fail when scenario baselines are stale, so that no PR can land with mismatched reference scores.
28. As a maintainer, I want each arena run to use process isolation per `(agent, scenario)` pair, so that one misbehaving agent cannot poison the whole eval.
29. As a maintainer, I want the project's PyPI distribution name to match the new repo identity, so that there is no residual hackathon-era naming in user-facing artifacts.
30. As a maintainer, I want the original hackathon and upgrade briefs preserved as archived documents, so that the design history is auditable but does not clutter the agent-author-facing docs.

### Reviewers and observers

31. As an observer, I want to read a Markdown leaderboard committed to the repo, so that I can see the current ranking without standing up infrastructure.
32. As a reviewer, I want a recorded run to be self-contained in a single folder, so that I can audit a claimed score by inspecting files rather than by re-running the simulation.
33. As a reviewer, I want each recorded run to identify which scenario and seed produced it, so that I can correlate a score claim with the conditions it was produced under.

### Determinism and replay

34. As a developer running the test suite, I want the determinism test to still pass after the pricing-fields refactor, so that I am confident the move from constants to state-fields preserves byte-for-byte replay behavior.
35. As a developer debugging a regression, I want the action log to still allow re-simulating any game from disk, so that the existing replay tool continues to work.
36. As a developer, I want scenario attachment via the API to be captured in the action log, so that an action-log replay reproduces the exact sequence including any mid-game scenario attaches.

## Implementation Decisions

### Modules to build

- **Scenario module.** Defines a `Scenario` protocol with one method, `apply(world, day) -> None`, plus a `NullScenario` (the default, does nothing) and a `load_scenario(dotted_path)` helper that imports a module and returns its `Scenario` subclass. No file I/O, no RNG. Lives under the world package as a peer of `events.py`. This is the contract every scenario obeys.
- **Recorder module.** Writes the three artifacts of a recorded run (metadata file, states JSON-lines, final snapshot). Has one method called from the daily step (`record_step`) and one called when the world is reset or the run is otherwise finalized (`finalize`). Owns the run folder allocation and naming. Lives under the world package as a peer of `action_log.py`. The existing `ActionLog` continues to own its own JSON-lines file; the recorder is additive, not a replacement.
- **Scenarios package.** Top-level directory containing one Python file per shipped scenario. The v1 set is three: a baseline (null scenario for seed 42, kept as a class so the CLI surface is uniform), a grid-stress scenario combining sustained low-wind weeks with a heatwave cluster, and an economy-stress scenario combining a fuel shock, a crude-price collapse, and a regulatory tightening. Each scenario class declares its seed as a class attribute.
- **Arena package.** Three modules. The runner orchestrates `(agent, scenario)` pairs as subprocesses, capturing per-pair results. The leaderboard takes a list of results and produces a ranked Markdown table via mean-rank aggregation, with ties broken by mean raw score. The results module owns the dataclass and the JSON I/O for the result schema. The runner has a CLI invokable as `python -m arena.runner`.

### Modules to modify

- **World state.** Add ten new mutable fields for prices and rates: crude price, refined price, grid retail price, grid export price, industrial revenue per day, commercial revenue per resident per day, daily tax per capita, blackout penalty hour, and a per-fuel-type dict for plant fuel costs. Plus a weather-overrides dict (string-keyed floats) for transient per-hour clips, and a scenario-trace list to record what the scenario did each day. Defaults match the existing module-level constants.
- **World reset.** Initialize the new state fields from their constant defaults. The reset path takes an optional scenario instance; when omitted, the world runs with `NullScenario`. The active scenario is stored on the world object as `self.scenario`.
- **Day loop.** Call the scenario's `apply` method once per day, after the expiry pass over finite-duration events and before the stochastic event sampler. This means a scenario-injected heatwave automatically suppresses a stochastic heatwave roll on the same day via the existing "is this event already active" guard.
- **Weather module.** After computing the AR(1) update for cloud factor and wind speed, consult the state's weather-overrides dict; if a key for the relevant observable is present, the override value wins. The override values are also written into the recorded state so replay viewers can see them.
- **Pricing-related modules.** Every read site for the ten promoted fields now reads from world state instead of importing a constant. Specifically: subsurface reads the crude price; economy reads the refined price and the fuel-cost dict; pricing helpers read the industrial and commercial rates; population reads the daily tax per capita; power reads the blackout penalty and both grid prices. The module-level constants remain in their original files as the defaults that initialization reads from; they are no longer the runtime source of truth.
- **API surface.** Three new endpoints. A POST endpoint to attach a scenario mid-game by dotted module path; a GET endpoint to report the currently-attached scenario; a GET endpoint to report the current run folder identifier so the UI can show it. The existing reset endpoint's body grows one optional field, the scenario dotted path. All new endpoint calls write to the action log.
- **UI.** Add a file picker for loading a run folder via the browser File API; parse the states JSON-lines client-side; toggle between live mode and replay mode based on whether a run is loaded. Add play, fast-play, and pause controls that drive an auto-advance timer at either 500 or 250 milliseconds per day. In live mode the play timer calls the step endpoint; in replay mode it advances an in-memory index. Keyboard shortcuts mirror the buttons. The existing Next-Day, Reset, build, drill, and survey controls are unchanged.
- **CLI.** The single-game evaluation script gains an optional `--scenario` flag accepting a dotted module path. The replay subcommand reads the scenario from the run folder's metadata.
- **Packaging.** Update the PyPI distribution name in the project metadata file. The Python package directories (world, agents) keep their current names; no imports change. Delete the stale egg-info directory.

### Documentation files

Seven Markdown files live at the repository root: a README that explains the project in 60 seconds and points to everything else; a rules file covering all game mechanics, formulas, build catalog entries, balance, and scoring (extracted and rewritten from the existing hackathon and upgrade briefs but targeted at agent authors arriving cold); an API reference enumerating every endpoint with request/response examples and error codes; a scenarios guide describing the protocol, the override taxonomy, and the v1 shipped scenarios with their stress profiles; a contributors' guide covering the test/format/lint loop, the PR-as-submission flow for community agents, and where to put scenarios; an MIT license file; and a leaderboard file regenerated by the arena leaderboard tool. The original hackathon and upgrade briefs move into a new archive subdirectory under `docs/`. Existing internal agent-skill documentation under `docs/agents/` stays where it is.

### Baselines

Each shipped public scenario has a committed baseline file under `baselines/`, named to encode both the scenario and the seed. The Makefile gains a target that regenerates every baseline by running the scripted reference agent against every public scenario. CI calls this target on PRs that touch the world or scripted-agent code and fails the build if any committed baseline file is stale. The hidden evaluation suite uses different seeds and is regenerated by the maintainer offline.

### Recording on every game

Every world session, whether driven by a remote agent or by the UI's interactive play, writes its run folder. The session field in the metadata file distinguishes the two. The runs directory is gitignored. A reset call finalizes the in-progress run (the final-snapshot file is written) and allocates a fresh run identifier for the next game; no run is destroyed by a reset.

### Score formula and arena aggregation

The score formula is unchanged: population term, treasury-delta tanh term, and renewable-share term, weighted 0.5/0.4/0.1. The arena's cross-scenario aggregation is mean rank: for each scenario, agents are ranked by raw score; an agent's arena rank is the mean of its per-scenario ranks; ties break by mean raw score, then submission timestamp.

### Submission protocol

Community agents are added to a `community/` subdirectory under the agents package via pull request. Each submission is one Python file containing a class that satisfies the agent base interface, with a header docstring naming the author and summarizing the approach. The leaderboard file is regenerated on merge and committed manually as part of the merge or a follow-up. The existing personal-workspace `submit/` directory keeps its current purpose as an iteration scratchpad and is not where final submissions live.

## Testing Decisions

The repo's existing pytest-driven test layout is preserved; new tests live alongside the modules they exercise. A good test in this codebase asserts external behavior — what callers see — not internal implementation. The existing test suite is exemplary on this point: dispatch tests assert per-source kilowatt-hours and balance states given a fixture world, not how the merit-order loop is implemented; determinism tests assert byte-identical replay across runs without inspecting RNG internals; economy tests assert revenue and CO2 figures given a hand-constructed tile set.

The PRD requires tests on the following surfaces:

- **Scenario module.** Unit tests cover the loader (valid dotted path returns an instance, invalid path raises a clear error, a module with no scenario subclass raises a clear error) and the null scenario (calling apply on it does not mutate any state). Prior art: the existing API loader tests for the agent resolution path in `evaluate.py`.
- **Recorder module.** Unit tests cover the on-disk schema produced after one and after many steps; that the metadata file contains seed, scenario name, session marker, and started-at timestamp; that the final-snapshot file is written exactly once per run regardless of how many times finalize is called. Prior art: the action-log tests verify the JSON-lines shape of the existing action log.
- **Leaderboard module.** Pure-function tests: known input result sets produce known rank tables; ties break correctly; agents missing from one scenario are handled with a defined policy (excluded from the missing scenario's rank, included in the others). Prior art: the existing scoring tests are pure-function tests of the score helper.
- **Pricing refactor regression tests.** Assert that with default world state values (i.e. matching the original module-level constants), the dispatch, end-of-day fuel-cost accrual, refinery revenue, industrial and commercial revenue, tax revenue, and blackout penalty accrual produce byte-identical figures to the pre-refactor behavior. Prior art: the existing dispatch and economy tests; this is a regression-only pass, not new mechanics testing.
- **Each shipped scenario.** A small test per scenario asserts that the scenario's documented overrides fire on the documented days and clear on the documented days. The tests drive a few days of simulation and read the state-side overrides or active-events list directly. Prior art: the existing events-module tests, which probabilistically sample and then assert state shape.
- **Arena runner integration.** One end-to-end test runs the scripted agent against the baseline scenario through the runner CLI and asserts the result row matches the committed baseline file (population, treasury delta, renewable share). This test is intentionally slow and may be marked accordingly. Prior art: the existing API smoke test exercises the full FastAPI surface in one test.

Test coverage for the UI is not required by this PRD. The frontend has no test runner today and standing one up is out of scope.

The determinism contract is preserved: the existing determinism test (scripted agent on seed 42 twice, byte-identical state) must continue to pass after the pricing refactor and after the introduction of the scenario hook in the day loop. Adding the new mutable state fields and adding the scenario `apply` call (which is a no-op when `NullScenario` is attached) must not change the byte trace of a baseline-seed run.

## Out of Scope

This PRD does not deliver any of the following. They are explicitly deferred to future work:

- **Multi-tenant arena infrastructure.** No simultaneous multi-agent sessions; the world is single-tenant per process, as today.
- **LLM token-budget enforcement.** The original brief mentions a 500,000 token budget per game; this PRD does not add token-counting middleware. Participants self-enforce.
- **Hosted web leaderboard.** The leaderboard is a committed Markdown file regenerated by a CLI tool; no web service is introduced.
- **Tax mechanics as a first-class concept.** A new `tax_rates` dict on world state is *not* added. Scenarios that want to simulate a tax do so by scaling the corresponding revenue-rate field on world state, accepting that the visible accounting is "revenue went down" rather than "a tax line item appeared."
- **Parallel arena runs.** The runner executes pairs sequentially via subprocesses. A `--parallel` flag is not added in v1.
- **Mid-day state snapshots.** The recorder writes one snapshot per simulated day, not per intra-day action. Intra-day actions remain visible only through the action log.
- **A dedicated scenario RNG stream.** Scenarios are deterministic Python functions of `(world, day)` and consume no random numbers in v1. If a future scenario needs stochasticity, a fourth child of the master seed sequence will be added then, preserving the existing three streams.
- **External-repository agent submissions via manifest.** Community agents must be submitted as files in the agents/community directory; the PRD does not introduce a registry pointing to off-repo code.
- **Renaming the Python package directories.** The `world` and `agents` Python packages keep their current names; only the PyPI distribution name in project metadata changes.
- **A new game mechanic.** This PRD adds no new tile types, no new event types, no new dispatch behavior, no new economic concepts. The existing world is wrapped, not extended.
- **A frontend test runner.** UI behavior is verified by manual play during code review of the relevant PRs.

## Further Notes

### On the live-vs-replay split

The clean separation between live mode (talks to the server, polls state, drives steps) and replay mode (reads files in the browser, advances an in-memory index, no server contact) is the central UX simplification. It means the recorder produces a self-contained artifact that survives the server being shut down, and it means the UI does not need new endpoints for replay. A side benefit: anyone can email a zipped run folder to a colleague who can open it in their browser without standing up the server.

### On the action log versus the state log

These two artifacts now coexist in every run folder. The action log captures every API call (including mid-game scenario attaches) and is the substrate the replay-by-re-simulation tool uses to verify determinism. The state log captures end-of-day world state and is the substrate the UI replay viewer uses to render a finished game. They serve different audiences and should not be conflated. Both are committed to the run folder; the recorder owns the state log, the action log continues to own itself.

### On the constants-to-state refactor

This is the most invasive part of the PRD by line count and the part most likely to ship a subtle regression. The mitigation is a complete pass through every read site for each promoted constant, accompanied by the regression tests called out above. The constants themselves are not deleted from their original files; they remain as the default values that world reset initializes the state fields from. This dual-source-of-truth is intentional: it preserves the readability of the original modules (where the canonical default is still co-located with the equation) and allows the world-reset routine to be the single point where defaults flow into runtime state.

### On the choice of mean rank for arena aggregation

Mean rank is scale-invariant — a scenario with raw scores clustered around 0.5 does not dominate a scenario with raw scores clustered around 1.8. This matters because stress scenarios will produce systematically different score distributions than the baseline scenario, and a naive mean-of-scores would let one outlier scenario decide the leaderboard. Tie-breaking by mean raw score (the underlying signal) and then by submission timestamp gives a complete deterministic ordering.

### On future extensions

Several capabilities deferred here have natural growth paths. Tax mechanics, if community demand emerges, are roughly 20 lines of state schema plus consultation in the relevant pricing helpers. A scenario RNG stream is one extra `spawn` call on the master seed sequence. Parallel arena runs are a thread pool over the existing subprocess invocations. A hosted leaderboard is a Markdown file plus a deploy hook. The architectural decisions in this PRD — single hook scenarios, state-side prices, file-based recording — were made so that each of these is additive when needed and absent when not.

### On the project's hackathon origin

The repository's structure is a credit to the original brief: every formula in the brief appears as a named function in code with matching variable names, every module is readable in 10 minutes, and the test suite asserts external behavior rather than internal mechanics. The open-source arena is not a rewrite. It is a small, targeted addition to a codebase already shaped for community contribution.
