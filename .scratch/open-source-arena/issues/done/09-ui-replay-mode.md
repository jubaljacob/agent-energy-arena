# 09: UI replay mode — file picker, JSON-lines parse, scrub, step back

Status: ready-for-agent
Category: enhancement
Verification: AFK code + HITL play test (see "Manual verification protocol" below)

## Parent

`.scratch/open-source-arena/PRD.md`

## What to build

Add a "load run" file picker to the UI that reads a run folder via the browser File API and renders the recorded game without server contact. Parsing happens client-side from `states.jsonl` and `metadata.json`. The existing live-play render path is unchanged; replay is a separate render mode toggled by whether a run is loaded.

Replay mode supports:
- Day-by-day scrubbing along a timeline.
- Step forward to the next day; step back to the previous day for before/after comparisons.
- Display of the metadata identifying scenario, seed, session marker, and run id.

The UI does NOT need new server endpoints for replay; everything is file-based in the browser.

## Acceptance criteria

- [ ] A file picker in the UI accepts a run folder and switches the UI to replay mode.
- [ ] In replay mode, the UI renders end-of-day world state from `states.jsonl` without contacting the server (no `fetch()` calls fire while scrubbing).
- [ ] The user can step forward and step back through days.
- [ ] Metadata (scenario, seed, session marker, run id) is visible while in replay mode.
- [ ] Live mode is unchanged when no run is loaded; the existing Next-Day, Reset, build, drill, and survey controls still work.
- [ ] A zipped run folder shared between machines opens and renders the same game on the receiving side without a server (verified by the manual protocol below).
- [ ] The PR description includes a filled-in copy of the **Manual verification protocol** section below, with each step ticked off and any screenshots or console-log excerpts that the reviewer would need to repeat the test.

## Agent Brief

> *This was generated by AI during triage.*

**Category:** enhancement
**Summary:** Add a client-side replay mode to the UI that reads a run folder (`metadata.json` + `states.jsonl`) via the browser File API and renders any day on demand without server contact.

### Current behavior

- The UI is a single-page app served by FastAPI: three static files (`world/ui/index.html`, `world/ui/app.js`, `world/ui/style.css`). No JS bundler, no framework, no package.json. Plain DOM + canvas + SVG, wrapped in one IIFE in `app.js`.
- The render entry point is `tick()` near the bottom of `app.js`. It does `fetch("/state")`, parses the response, populates the module-level globals (`tiles`, `wells`, `summary`, `treasury`, `worldDims`, `activeEvents`, `historicalEvents`, …), updates the top-bar metrics, and calls the per-section render helpers (`renderPowerChart`, `renderPowerMargin`, `renderPlantList`, `renderWells`, `renderRefineries`, `renderFinance`, `renderEvents`, `drawGrid`, optionally `refreshRevealed` + `renderSubsurface`).
- Every mutating action (build, demolish, survey, drill, control, step) calls `tick()` itself; there is no `setInterval(tick)` polling loop.
- The `GET /state` payload is exactly `World.state_dict()` (see `world/api.py` — the endpoint is `return app.state.world.state_dict()`). The recorder writes the same `state_dict()` into each line of `states.jsonl` (see `world/recorder.py::Recorder.record_step` — each line is `{day, state: world.state_dict(), summary: world.state.today_summary_so_far}`). **This is the key reuse hook: a `states.jsonl` line's `.state` field has the identical shape as a `/state` response, so the existing render helpers work unchanged in replay mode.**
- `metadata.json` is written once at run construction (`Recorder._write_metadata`) and contains: `run_id`, `seed`, `scenario` (dotted path or null for `NullScenario`), `session`, `started_at`.
- A typical run folder also contains `actions.jsonl` (owned by `world.action_log.ActionLog`) and `final.json` (owned by the recorder). The replay UI only needs `metadata.json` + `states.jsonl`; the other files can be ignored.

### Desired behavior

**Mode switching.** Introduce an explicit mode flag (e.g. `let mode = "live"`). When `mode === "live"`, all existing behavior is preserved — `tick()` polls `/state`, mutating buttons hit their endpoints, etc. When `mode === "replay"`, the live `tick()` path is short-circuited (or never called) and rendering is driven from an in-memory array of parsed `states.jsonl` lines plus a cursor index.

**Load button + file picker.** Add a "Load run" control to the top bar (or a nearby header region). It opens a directory picker via `<input type="file" webkitdirectory directory multiple>`. The change handler:
1. Filters the `File[]` for entries whose `webkitRelativePath` ends in `metadata.json` or `states.jsonl`.
2. Reads both files as text (FileReader → `text()` on the Blob).
3. Parses `metadata.json` as a single object and `states.jsonl` line-by-line (skip blank lines, parse each non-empty line as JSON).
4. Stores the parsed metadata + the array of parsed state lines on a module-level variable.
5. Sets `mode = "replay"`, resets the cursor to the last day available, calls a new `renderReplayFrame()`.
6. Shows the metadata (scenario / seed / session / run_id) in a visible badge.

**Render in replay mode.** Add `renderReplayFrame(index)`. It takes the cursor index, grabs `loadedStates[index]`, and populates the same module-level globals from `entry.state` that `tick()` populates from the `/state` response. Then it calls the same render helpers `tick()` already calls. Constraint: it MUST NOT call `fetch()`. If a render helper today calls `fetch()` synchronously, refactor that helper so the network call is in `tick()` only and the helper takes the data as an argument. (As of the current code only `tick()`, the mutating handlers, `refreshRevealed`, and `loadCatalog` issue `fetch()`. `loadCatalog` is allowed in replay mode if it runs once at app start before any run is loaded — the catalog endpoint is static and the replay viewer needs the catalog to render the build palette and cost lookups.)

**Timeline + scrubbing.** Add a timeline control visible in replay mode: a range slider with `min=0`, `max=loadedStates.length - 1`, and a numeric display "Day X / Y". Wire its `input` event to `renderReplayFrame(cursor)`. Add "← Step back" and "Step forward →" buttons that move the cursor by ±1 with bounds clamping. Keyboard shortcuts (left/right arrow) mirror the buttons but only when replay mode is active and no input element has focus.

**Exiting replay.** A "Close replay" button switches `mode` back to `"live"`, clears `loadedStates`, hides the timeline, restores the live top-bar metrics by calling `tick()` once.

**Empty / partial runs.** If `states.jsonl` is empty, show "no recorded days" and stay in replay mode with the timeline hidden. If `metadata.json` is missing but `states.jsonl` parses, fall back to "(metadata unavailable)" placeholders and still render the days. If `states.jsonl` lines fail to JSON-parse, skip the bad lines and surface a toast with the count.

**Cross-machine zip support.** The browser File API returns directory entries with `webkitRelativePath` like `<root>/states.jsonl`. A zipped folder unzipped on a different machine has the same shape (the root folder name may differ but the relative paths inside are preserved). The change handler's filename match must use `endsWith("/states.jsonl")` and `endsWith("/metadata.json")` (or the bare filename) — NOT a fully-qualified path match.

### Key interfaces

- **Run folder schema** — `metadata.json` is one JSON object with at least `{run_id, seed, scenario, session, started_at}` (current keys; the recorder may grow more, so don't assume a closed set — just read the four metadata fields needed for the badge and ignore the rest). `states.jsonl` is line-delimited JSON; each non-empty line is `{day: int, state: object, summary: object}`. The `state` object is structurally identical to the `/state` endpoint response (both come from `World.state_dict()`).
- **Render helpers in `app.js`** — `drawGrid`, `renderPowerChart`, `renderPowerMargin`, `renderPlantList`, `renderWells`, `renderRefineries`, `renderFinance`, `renderEvents`, `renderSubsurface`, plus the top-bar metric writes (`els.day.textContent = …`). These should all be callable from a replay path that has populated the module-level globals from a recorded state, without modification beyond making sure they read from those globals (most already do).
- **Mode-gated controls** — the mutating buttons (Next Day, Reset, build palette items, drill, survey, well/refinery controls) should be visibly disabled or hidden in replay mode. Their click handlers should also early-return if `mode === "replay"` so a keyboard shortcut can't bypass the visual disable.
- **No new server endpoints.** Replay is entirely client-side. Do not add anything to `world/api.py`.

### Out of scope

- The play / fast-play / pause auto-advance timer and the mid-game scenario attach control — that's issue 10. This issue only ships manual scrubbing (slider + step buttons + arrow keys).
- A frontend test runner. UI behavior is verified by the manual protocol above; standing up Vitest / jsdom / Playwright is explicitly out of scope per the parent PRD ("Out of Scope" section).
- Visual redesign / restyling. New replay controls should follow the existing CSS conventions in `world/ui/style.css` (side panels, balance badge styling, button styling) — match, don't reinvent.
- Reading `actions.jsonl` or `final.json`. Replay is driven from `states.jsonl` only. The action log is for the re-simulation determinism tool, not the visual viewer.
- Server-side endpoints for replay. The PRD explicitly states "everything is file-based in the browser."
- Multi-run comparison / diffing. One loaded run at a time.

### Notes for the AFK agent

- **You will not be able to test this in a browser.** Read the existing `app.js` carefully, reuse the render helpers, and reason from the recorder/state shapes documented above. The reviewer is responsible for the manual play test; your job is to make that test pass on the first attempt by being thorough on the code side.
