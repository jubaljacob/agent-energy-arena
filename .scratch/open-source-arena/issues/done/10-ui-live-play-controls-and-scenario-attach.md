## What to build

Add play, fast-play, and pause controls to the UI that drive an auto-advance timer:
- Play: 500 ms per day, calls the existing step endpoint each tick (live mode) or advances the in-memory index (replay mode).
- Fast-play: 250 ms per day.
- Pause: stops the timer; current day stays on screen for inspection.
- Keyboard shortcuts mirror the buttons.

Add a UI control to attach a stress scenario mid-game by dotted path. The control calls the `POST /scenario` endpoint from issue 04 and displays the currently-attached scenario name via `GET /scenario`. Existing Next-Day, Reset, build, drill, and survey controls remain unchanged.

## Agent Brief

> *This was generated by AI during triage.*

**Category:** enhancement
**Summary:** Add an auto-advance timer (play / fast-play / pause) that works in both live mode (calls `POST /step`) and replay mode (advances the in-memory cursor), plus a dotted-path input that attaches a scenario via `POST /scenario` and a readout that polls `GET /scenario` for the currently-attached name.

### Current behavior

- The UI is a single-page app served by FastAPI: `world/ui/index.html` + `world/ui/app.js` + `world/ui/style.css`, plain DOM + canvas + SVG inside one IIFE. No JS bundler, no framework.
- The only day-advance control today is `#next-day`. Its click handler awaits `POST /step` then calls `tick()` to repopulate from `GET /state`.
- The render path is `tick()` (see issue 09's brief for the full call graph). Every mutating handler awaits its endpoint and then calls `tick()` itself — there is no polling loop.
- The two new server endpoints this issue depends on already exist in `world/api.py`:
  - `GET /scenario` returns `{dotted_path: str | null}`. `null` means `NullScenario` is attached (i.e. no stress scenario).
  - `POST /scenario` accepts `{dotted_path: str}` and loads the scenario via `world.scenario.load_scenario`. On success returns `{ok: true, dotted_path}`. On invalid path returns HTTP 400 with `{detail: "..."}` and the action log records the failure.
- Issue 09 (which this issue depends on) introduces a `mode` flag with values `"live"` and `"replay"`, a `loadedStates` array, and a `cursor` index. The replay scrubbing controls move the cursor manually; this issue adds the auto-advance timer that moves it (or calls `/step`) on a setInterval.
- v1 scenarios live as `scenarios.baseline.Baseline`, `scenarios.grid_stress.GridStress`, `scenarios.economy_stress.EconomyStress` (dotted paths the user might type into the input).

### Desired behavior

**Auto-advance timer.** Add three buttons to the top bar near the existing `#next-day`: ▶ Play, ⏩ Fast, ⏸ Pause. A single module-level `playTimer` holds the `setInterval` handle (or `null` when paused). Starting the timer cancels any existing one first (no overlapping intervals). The interval period is 500 ms for Play and 250 ms for Fast.

On each timer tick:
- If `mode === "live"`: invoke the same logic as the Next-Day button (await `POST /step`, then `tick()`). If `/step` returns non-OK (e.g. game over per `world.config.GAME_DAYS`), pause the timer and surface a toast.
- If `mode === "replay"`: increment `cursor` by 1 and call `renderReplayFrame(cursor)`. If `cursor` reaches `loadedStates.length - 1`, pause the timer.

**Re-entrancy guard for live mode.** `POST /step` is async and can take longer than 250 ms on slow hardware. Guard against overlapping requests: if the previous tick's `/step` is still in flight when the next interval fires, skip that tick (don't queue, don't await two in parallel). A simple `let stepInFlight = false` flag is sufficient.

**Pause.** Pause cancels the timer, leaves the displayed day exactly where it is. Pause is also called automatically on: mode switch (live↔replay), Reset click, error responses, and `cursor` reaching the end of replay states. A second click on Play or Fast while the same mode is already running is a no-op (don't restart and skew the cadence).

**Keyboard shortcuts.** Document them in the UI (a small `<span>` next to the buttons or a tooltip is fine). Suggested bindings, but use whatever doesn't collide with existing browser/UI shortcuts:
- `Space` — toggle Play/Pause
- `Shift+Space` — toggle Fast/Pause
- (issue 09 already claimed `←` / `→` for replay step)

Shortcuts only fire when no input element has focus (`document.activeElement.tagName !== "INPUT"` and not a textarea).

**Scenario attach control.** Add a small panel (in the events tab or top bar — pick one with available real estate) containing:
1. A read-only display of the currently-attached scenario, populated from `GET /scenario`. Show `"(none)"` when `dotted_path` is null. Refresh on app start, after `POST /scenario` succeeds, and after every Reset.
2. A text input + "Attach" button. The Attach handler:
   - Reads the input's value, trims whitespace.
   - POSTs `{dotted_path: <value>}` to `/scenario`.
   - On 2xx: surface a success toast, refresh the readout, call `tick()` to repopulate (the next day's render will reflect any state mutations the scenario performs on its first `apply`).
   - On 400: surface the server's `detail` string as an error toast; leave the existing scenario attached.
3. A "Detach" button (or quick-attach to `NullScenario`). Since there is no `DELETE /scenario` endpoint, "Detach" should be implemented as `POST /scenario {dotted_path: "world.scenario.NullScenario"}` — that's the canonical no-op scenario already exported from the world package. Verify by reading `world/scenario.py` for the actual `NullScenario` class location before wiring this; if the dotted path differs, use the one in the code.

**Disabled in replay mode.** The Attach button and its input are disabled (and visually grayed) when `mode === "replay"`. The readout still shows the metadata's `scenario` field from the loaded run instead of polling `GET /scenario`.

**Existing controls untouched.** Next-Day, Reset, build palette, drill, survey, well/refinery control sliders behave exactly as today. The only change to Reset is that it must call the new `pauseTimer()` helper before doing its existing work.

### Key interfaces

- **`POST /step`** — body `{}`. Response `{ok: true, day_completed: int}` on success. On end-of-game returns non-2xx; the timer should pause and surface a toast.
- **`GET /scenario`** — response `{dotted_path: string | null}`. `null` = `NullScenario`.
- **`POST /scenario`** — body `{dotted_path: string}`. 2xx response `{ok: true, dotted_path}`. 4xx response `{detail: string}` for invalid module paths or modules without a `Scenario` subclass.
- **Mode flag from issue 09** — `mode: "live" | "replay"`. Read-only from this issue's perspective; never write it from this issue's handlers (use the helpers issue 09 exposes — `replay.setMode()` or equivalent).
- **`tick()`** and **`renderReplayFrame(index)`** — existing render entry points; the timer dispatches to one or the other based on mode.

### Notes for the AFK agent

- **You cannot test this in a browser; the reviewer will.** Code defensively: re-entrancy guard, pause on every Reset and on every mode switch, clear the interval on the page-unload event (a single `window.addEventListener("beforeunload", pauseTimer)` is cheap insurance).
- The IIFE wrapper in `app.js` means new module-level state (`playTimer`, `playSpeed`, `stepInFlight`, `attachedScenarioPath`) can live next to the existing globals. Match the style of the surrounding code rather than introducing a class.
- A small `pauseTimer() / startTimer(ms)` helper pair keeps the call sites readable. Don't sprinkle `clearInterval` everywhere — funnel through one helper.
- The PRD's user stories 17–19 are the canonical statement of intent for this issue (Play at 0.5 s, fast-play at 0.25 s, pause holds the current day). Worth quoting in the commit message.
- Issue 09's brief documents the live-vs-replay split as a "separate render mode, not a hybrid". Honor that here too: the timer dispatches to one of the two existing render entry points, it does not blend them.
