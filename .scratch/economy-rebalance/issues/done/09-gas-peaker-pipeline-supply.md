# Gas peaker requires pipeline-to-refinery (dispatch-time)

Status: ready-for-agent
Category: enhancement

## Parent

[PRD: Economy & balance pass](../PRD.md)

## What to build

Couple the gas-peaker fleet to the oil loop: a gas peaker only dispatches if it shares a 4-connected pipeline network with at least one operational refinery. The check runs at dispatch time (each hour), so destroying a refinery mid-game immediately stops dispatching every peaker on its network. A peaker without refinery supply is treated identically to a `plant_failure` (zero output that hour), so downstream brownout/blackout accounting Just Works.

Implementation:

- New query function in `world/pipelines.py` — `peaker_supply(peaker_tile, tiles) -> bool` — that reuses the existing crude-routing network walker (no new graph representation).
- `power.dispatch` filters gas peakers without refinery supply before merit-order ordering, treating them as failed for that hour.
- Gas peaker catalog description is updated to surface the pipeline-to-refinery requirement.

## Acceptance criteria

- [ ] `pipelines.peaker_supply` returns True for a peaker sharing a contiguous pipeline path with an operational refinery.
- [ ] Returns False when the only connected refinery is non-operational.
- [ ] Returns False for a peaker on an isolated pipeline network.
- [ ] Returns False for a peaker not adjacent to any pipeline.
- [ ] A peaker with refinery supply dispatches normally; a peaker that loses pipeline connectivity dispatches zero kW that hour.
- [ ] The zero-output dispatch path flows through the same accounting as `plant_failure` (no new brownout/blackout code branch).
- [ ] Gas peaker catalog description mentions the pipeline-to-refinery requirement.
- [ ] New tests in `world/tests/` cover the four `peaker_supply` cases and the dispatch gating behavior.
- [ ] `make check` passes.

## Agent Brief

**Category:** enhancement
**Summary:** Gas peakers only dispatch when they share a pipeline network with an operational refinery, evaluated each hour at dispatch.

**Current behavior:**
Gas peakers can be placed and dispatched in isolation with no upstream pipeline dependency, decoupling them from the oil loop they were designed to consume from.

**Desired behavior:**
Each hour, before merit-order ordering, dispatch filters out any gas peaker that does not share a 4-connected pipeline network with at least one operational refinery. A filtered peaker behaves identically to a plant-failure event (zero output that hour) — downstream brownout/blackout accounting goes through the existing path. Destroying a refinery mid-game causes every peaker on its network to stop dispatching immediately on the next hour. The peaker's catalog description states the pipeline-to-refinery requirement.

**Key interfaces:**
- A new query in the pipelines module: `peaker_supply(peaker_tile, tiles) -> bool`, reusing the existing crude-routing network walker (no new graph representation).
- The hourly dispatch path — calls `peaker_supply` for each gas peaker and routes failures through the existing plant-failure zero-output handling.
- The gas peaker entry in the tile catalog — description mentions the pipeline-to-refinery requirement.

**Acceptance criteria:**
- [ ] `peaker_supply` returns True when the peaker shares a contiguous pipeline path with at least one operational refinery.
- [ ] Returns False when the only connected refinery is non-operational.
- [ ] Returns False for a peaker on an isolated pipeline network (no refinery reachable).
- [ ] Returns False for a peaker with no pipeline adjacency at all.
- [ ] A peaker that loses connectivity (refinery destroyed) outputs zero kW on the immediately following hour without any code branch new to brownout/blackout accounting.
- [ ] Gas peaker catalog description mentions the pipeline-to-refinery requirement.
- [ ] `make check` passes.

**Out of scope:**
- Multi-fuel peaker behavior (only gas peakers exist).
- Pipeline construction, capex, or routing logic.
- A persistent "peaker degraded" UI state — the dispatch hour-zero is sufficient signal.
